# 架构与模块设计

## 1. 前端分层

前端入口是 `src/App.tsx`。它组合全局状态、主布局、设置弹窗、关闭确认弹窗和 Chat/Ops 两个主视图。具体业务逻辑主要下沉到 hooks：

- `useChat`：发送消息、流式事件、上下文压缩、RAG 召回、Agent run、工具执行续写、图片附件上传。
- `useConversations`：普通会话、项目会话、归档会话、会话模型绑定和预览消息加载。
- `useModel`：模型配置、连接测试、当前模型、embedding 配置。
- `useProjects`：项目列表、项目目录创建、项目会话映射。
- `useRagFiles`：RAG 文件列表、索引状态、召回。
- `useMcp`：MCP server 配置、连接、断开、工具刷新。
- `useSkills`：Skills 同步、本地 Skills、启用状态和 app data 下的 `temp/` 目录。
- `useWorkspace` / `useMemory`：本地条目和长期记忆管理。
- `useObservability`：观测 span 拉取、trace 分组、时间线展开状态。

`src/api.ts` 是前端统一后端调用封装。组件不直接写 `invoke`，而是调用这里的类型化函数，保持 command 名称、参数和返回类型集中管理。

## 2. UI 模块

主要界面分为：

- `ChatPane`：聊天消息、模型选择、RAG 文件条、图片附件入口、运行时面板。
- `Sidebar`：普通会话、项目入口、项目会话、Chat/Ops 主视图切换。
- `WorkspaceGrid`：笔记、提示词、记忆等本地对象管理。
- `SettingsModal`：主题、记忆、模型、embedding、归档、观测、Skills、MCP、环境配置。
- `SettingsArchiveTab`：归档列表和归档消息预览。图片消息通过 `renderMessageContent` 和 `ImageAttachmentMessage` 渲染，附件根目录来自归档会话的 `project_path`，没有项目路径时回退到 `skills.tempDir`。
- `OpsPanel`：SSH 服务器配置、连接测试、上传文件、交互式终端。

## 3. 后端分层

Rust 后端由 `src-tauri/src/lib.rs` 装配：

- 初始化 Tauri plugin、系统托盘和 app data 目录。
- 打开主业务库、Agent 运行时库、观测库。
- 创建 `AppState`，包含 `Database`、`RuntimeStore`、`ObservabilityPipeline`、`McpClientManager` 和 Ops SSH session 状态。
- 注册所有 Tauri command。

后端模块边界：

| 模块 | 职责 |
| --- | --- |
| `db.rs` | 主业务数据读写、FTS、RAG、memory、conversation、ops server 等 |
| `runtime.rs` | Agent run、step、tool_call 持久化 |
| `observability.rs` | span 数据结构、sink trait、SQLite sink |
| `llm.rs` | 模型请求、流式事件解析、embedding |
| `mcp.rs` | MCP 连接、工具列表、工具调用 |
| `agent_runner.rs` | 工具定义、XML tool_call 解析、参数校验 |
| `skills.rs` | GitHub Skills 同步和本地 Skills 列表 |
| `models.rs` | 前后端共享的序列化模型 |
| `error.rs` | 统一错误类型 |

## 4. Command 设计

Tauri command 按领域分组：

- 本地条目：`list_items`、`search_items`、`create_item`、`update_item`、`delete_item`
- 模型配置：`list_model_configs`、`save_model_config`、`test_llm_connectivity`
- MCP：`list_mcp_servers`、`connect_mcp_server`、`call_mcp_tool`
- 对话：`list_conversations`、`append_message`、`archive_conversation`
- RAG：`index_rag_file`、`search_rag_context`
- 记忆：`list_memories`、`search_memories`、`create_memory`
- Skills：`sync_anthropic_skills`、`list_local_skills`
- 模型调用：`chat`、`chat_stream`
- Agent 运行时：`create_agent_run`、`record_agent_step`、`execute_agent_tool_call`
- 项目文件与附件：`list_project_files`、`read_project_file`、`write_project_file`、`save_chat_image_attachment`、`read_chat_image_attachment`
- Ops：`list_ops_servers`、`save_ops_server`、`test_ops_ssh_connection`、`upload_ops_file`、`start_ops_ssh_session`
- 观测：`list_observability_spans`、`clear_observability_spans`
- 系统窗口：`minimize_to_tray`、`show_app_window`、`quit_app`

## 5. 状态与并发模型

`AppState` 中的 Rust 状态使用 `tokio::sync::Mutex` 包装：

- `db`：主业务 SQLite connection。
- `runtime`：运行时 SQLite connection。
- `observability`：观测 pipeline。
- `mcp`：MCP client manager，持有活动 session。
- `ops_ssh_sessions`：交互式 SSH session 句柄。

这种设计简单可靠，适合桌面单用户应用。代价是同一类资源的并发访问会串行化，但能降低 SQLite connection、外部进程和 SSH session 状态复杂度。

## 6. 前后端事件

模型流式输出不通过 command 返回完整字符串，而是：

1. 前端生成 `request_id`。
2. 调用 `chat_stream`。
3. Rust 后端通过 Tauri event `chat-stream` 发送事件。
4. 前端用 `listen<ChatStreamEvent>` 过滤当前 `request_id`。

事件类型包括：

- `delta`：普通内容增量。
- `reasoning_delta`：reasoning/thinking 增量。
- `error`：流式请求错误。
- `done`：流式结束。

Ops 交互式 SSH 使用 `ops-ssh` 事件返回 `ready`、`data`、`error` 和 `closed`。

## 7. 依赖方向

推荐依赖方向：

```text
components -> hooks -> api -> tauri commands -> backend modules
```

跨层标识必须显式传递。尤其是 `project_path`、`conversation_id`、`model_config_id`、`trace_id`、`tool_call_id` 和图片附件根目录，不应该通过显示名或隐式全局状态猜测。
